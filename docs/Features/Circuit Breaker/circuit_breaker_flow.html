Circuit Breaker Integration

order-easy microservices project — complete guide & flow

What is Circuit Breaker?

Imagine calling someone on the phone. They don't pick up. You try again. And again. You're stuck in a loop wasting time.

Circuit Breaker says: "Stop trying after a certain number of failures. Give an instant fallback response. Try again after some time."

In our project, order-service calls inventory-service via Feign (synchronous HTTP). If inventory-service is down, Feign keeps waiting and the user hangs forever — or worse, the system crashes under load.

Circuit Breaker makes it fail-fast — instant 503 JSON response instead of hanging. We used Resilience4j library for this.

The 3 States — Core Concept
CLOSED
Normal mode. All requests pass through. CB is just watching failure rate.
OPEN
Failure threshold crossed. All requests blocked instantly. Fallback returned to client.
HALF-OPEN
After 10s wait, allows 3 test calls. If pass → CLOSED. If fail → OPEN again.

When Does Circuit OPEN?
failureRateThreshold = 50% → if 5 out of last 10 calls fail → circuit OPENS → blocks all traffic → waits 10 seconds
↓
after waitDurationInOpenState = 10 seconds...
HALF-OPEN State
CB allows exactly 3 test calls (permittedNumberOfCallsInHalfOpenState=3). Testing if service recovered.
↓
all 3 test calls pass
Circuit → CLOSED
Normal traffic resumes. All Feign calls go through again.
any test call fails
Circuit → OPEN again
Wait another 10 seconds. Cycle repeats.
Step-by-Step Integration — What We Did
StepFileWhat We Did
Step 1 pom.xml (order-service + cart-service) Added spring-cloud-starter-circuitbreaker-resilience4j and spring-boot-starter-aop. AOP is required — without it, @CircuitBreaker annotation does nothing.
Step 2 application.properties Added Feign timeouts: connectTimeout=3000, readTimeout=5000. Without timeouts, Feign waits forever on a dead service — timeouts make failures detectable.
Step 3 application.properties Configured CB instances: inventoryServiceCB, cartServiceCB in order-service. productServiceCB in cart-service. Set failureRateThreshold=50%, slidingWindowSize=10, waitDuration=10s.
Step 4 ExternalServiceProxy.java (NEW) Created a separate Spring Bean to hold @CircuitBreaker methods. Proxy pattern is needed because Spring AOP cannot intercept self-calls within the same class (self-invocation problem).
Step 5 OrderService.java + CartService.java Updated to call ExternalServiceProxy instead of calling Feign clients directly. Business logic stays clean, CB logic is in the proxy.
Step 6 GlobalExceptionHandler.java (NEW) Added handler for ServiceUnavailableException. Returns structured JSON with HTTP 503 status — production-grade response instead of raw exception.
Why Circuit Breaker on Feign only — NOT Kafka?

Feign = synchronous. order-service WAITS for inventory-service to respond. If it's down, the thread hangs indefinitely. Circuit Breaker prevents this with fail-fast behavior.

Kafka = asynchronous. order-service publishes an event and moves on immediately — it does NOT wait. If delivery-service is down, the event stays in the Kafka topic and gets consumed when the service comes back up. Kafka handles resilience itself — no Circuit Breaker needed.

Rule: Circuit Breaker only on synchronous Feign calls. Never on Kafka producers or consumers.
